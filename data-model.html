Data Model Reference

ORBIT Admin
K-pop Management Platform

Complete data schema for the ORBIT admin system. Each entity below includes a TypeScript interface, a sample JSON object, a field reference table, and its relational links to other entities. This is a fully linked, relational model — data is designed to interact across domains.

Architecture

Relational Map

How the entities connect to each other at a high level. Every arrow represents a foreign key relationship or array of references.

// Core hierarchy
Band
  ├── members[]       → Member
  └── releases[]      → Release

Member
  ├── soloProfileId   → SoloProfile   (optional — only popular members)
  ├── contractId      → Contract
  ├── healthRecords[] → HealthRecord
  └── endorsements[]  → Endorsement

Release  (albums, EPs, singles — group or solo)
  ├── memberId        → Member        (optional — solo releases only)
  ├── salesRecords[]  → AlbumSales
  ├── chartEntries[]  → ChartEntry
  └── socialSnapshots[] → SocialSnapshot

ScheduleEvent
  ├── bandId          → Band
  ├── venueId         → Venue
  ├── memberIds[]     → Member        (for solo/sub-unit events)
  ├── releaseId       → Release       (if event is a release showcase)
  └── incidentIds[]   → Incident      (disruptions linked to this event)

Incident
  ├── memberId        → Member        (optional — personal incidents)
  ├── venueId         → Venue         (optional — venue incidents)
  └── affectedEventIds[] → ScheduleEvent

Venue
  └── incidents[]     → Incident

Operations

Incident

A disruption event — either personal (affecting a member) or operational (affecting a venue or scheduled event). The category and subtype fields cover the full range of real-world K-pop disruptions.

Member (optional) Venue (optional) ScheduleEvent[]
📖

Naming note: "Incident" was chosen over "Life Event" because it covers both personal and operational disruptions neutrally, maps cleanly to a single data type, and is unambiguous in admin/ops contexts.

type IncidentCategory = 'personal' | 'venue' | 'operational';

type PersonalSubtype =
  | 'illness'            // cold, flu, fever
  | 'injury'             // sprain, fracture, vocal strain
  | 'mental_health_leave'
  | 'personal_leave'     // family bereavement, personal reasons
  | 'visa_issue'
  | 'travel_delay';

Ongoing physical and mental health tracking for a member, distinct from Incidents. While an Incident is a disruptive event, a Health Record is a longitudinal log — check-ups, fitness assessments, flagged concerns — managed by the agency's medical staff.

Game Layer · Visuals

Member Appearance

Drives the visual card rendered for each member. Assembled at runtime from a base face model plus expression and status overlay layers — no pre-saved combinations. A status like "sick" or "injured" is applied as a shared overlay that works across all 100 face models without additional art assets.

Member Incident (drives statusOverlay) HealthRecord (drives expression) 
type FaceExpression =
  | 'neutral'
  | 'happy'
  | 'tired'
  | 'focused'
  | 'stressed'
  | 'proud';

type StatusOverlay =
  | 'sick'       // desaturated + flush tint
  | 'injured'    // bandage asset layered on top
  | 'resting'    // soft vignette, muted palette
  | 'on_leave'   // greyed out, lock icon
  | null;         // no overlay — healthy

interface MemberAppearance {
  memberId:      string;
  faceModelId:   string;      // e.g. "face_034" — one of 100 base models
  gender:        'male' | 'female';
  expression:    FaceExpression;
  statusOverlay: StatusOverlay;
  updatedAt:     string;      // ISO 8601 — when state last changed
}

// Resolved at render time — never stored
interface RenderedCard {
  baseImageUrl:    string;   // "/faces/face_034.png"
  expressionUrl:   string;   // "/expressions/tired.png"
  overlayUrl:      string | null; // "/overlays/injured.png"
  cssFilter:       string | null; // e.g. "saturate(0.4)" for sick
}

}

Composition rule: expression and statusOverlay are shared assets applied on top of any base face. Adding a new status type requires one new overlay asset — not 100 new face variants. RenderedCard is computed by the UI layer and never persisted.

Game Layer · Ticker

World Event

A ticker entry generated when the player advances time. World events are the read-only log of everything that happened during a time increment — chart movements, sales updates, venue flags, member risk alerts. They are facts, not decisions. Anything requiring a player response spawns an InboxMessage instead.

Game Layer · Inbox

Inbox Message

A message requiring a player decision. Spawned either by a WorldEvent (e.g. a venue incident that needs a call) or by a member/label contact reaching out directly. Each message has a set of response options — the player picks one, and the outcome is applied to the relevant entities. Unanswered messages can expire or escalate.

WorldEvent (optional) Member (optional sender) Venue (optional) ScheduleEvent (optional) GameState 
type InboxSender =
  | 'member'
  | 'label'
  | 'promoter'
  | 'venue'
  | 'pr_team'
  | 'system';   // generated by game engine, no human sender

type MessageStatus =
  'unread' | 'read' | 'responded' | 'expired';

interface ResponseOption {
  id:          string;

}

Game Layer · Engine

Game State

The top-level record of where the player is in the game. Tracks the current in-game date, the default time increment, and summary counters. One GameState exists per save. Advancing time updates currentDate, resolves scheduled world events, runs member incident probability checks, and appends to the WorldEvent log.

Band WorldEvent[] InboxMessage[] 
type TimeIncrement = 'day' | 'week';

interface GameState {
  id:                string;
  bandId:            string;
  currentDate:       string;         // in-game date (ISO 8601)
  startDate:         string;         // band's debut date — game start
  timeIncrement:     TimeIncrement;  // player's preferred advance unit
  unreadInboxCount:  number;
  openIncidentCount: number;
  atRiskEventCount:  number;
  lastAdvancedAt:    string;         // real-world timestamp
  createdAt:         string;
  updatedAt:         string;
}

  "id":                "save_001",
  "bandId":            "band_01HX9K2M",
  "currentDate":       "2026-03-14",
  "startDate":         "2022-03-14",
  "timeIncrement":     "day",
  "unreadInboxCount":  2,
  "openIncidentCount": 1,
  "atRiskEventCount":  2,
  "lastAdvancedAt":    "2026-05-30T10:07:00Z",
  "createdAt":         "2026-01-01T00:00:00Z",
  "updatedAt":         "2026-05-30T10:07:00Z"
}
⚠️

Advance Time flow: When the player advances, the engine (1) increments currentDate, (2) resolves all deterministic scheduled events for that date, (3) runs probabilistic member incident checks, (4) appends WorldEvent entries to the ticker, (5) spawns any required InboxMessage records, and (6) updates GameState counters. The player sees the ticker update before they can act.
